# Member-Only Follow-Up Tasks

## 기준 상태

- 기준 브랜치: `codex/legacy-comment-cleanup-2026-04-22`
- 기준 시점: 2026-04-23
- 현재 정책:
  - 삭제/탈퇴 시 `mb_id`와 상태 정보만 유지한다.
  - 이름, 닉네임, 이메일, 휴대폰, 생년월일, IP, 인증 이력, 동의 로그, 임시 인증값은 비식별 처리 또는 삭제한다.
  - 관리자 UI에서는 탈퇴 회원 `mb_id`를 일부 마스킹해서 표시한다.

## 우선순위

### 1. QA 재현 절차 고정

- 목표:
  - 현재 수동 QA를 다음 작업자가 같은 순서로 다시 실행할 수 있게 만든다.
- 할 일:
  - Docker DB, Mailpit, PHP 서버 실행 명령 정리
  - 설치 초기화부터 회원가입, 이메일 인증, 비밀번호 재설정, 회원수정, 탈퇴, 관리자 회원 CRUD까지 재현 순서 정리
  - 실패 시 확인할 DB 필드와 테이블 목록 정리
- 완료 기준:
  - 새 작업자가 문서만 보고 로컬 QA를 다시 수행할 수 있다.

### 2. 스모크 자동화 추가

- 목표:
  - 수동 QA 중 반복되는 핵심 경로를 최소한의 자동 체크로 묶는다.
- 할 일:
  - PHP 또는 쉘 기반 스모크 스크립트 초안 작성
  - 삭제/탈퇴 후 개인정보 최소 보존 규칙 검증 추가
  - `sql_table_exists()` 회귀 여부를 검증하는 간단한 체크 포함
- 완료 기준:
  - 핵심 계정 흐름과 삭제 정책 회귀를 한 번에 확인할 수 있다.

### 3. 관리자 노출 최소화 추가 정리

- 목표:
  - 내부 식별은 유지하되 관리자 화면에서 불필요한 원문 노출을 더 줄인다.
- 할 일:
  - 탈퇴 회원 상세 화면에서 추가 노출 필드가 없는지 재검사
  - 관리자 대시보드, 검색 결과, 로그성 문구에서 탈퇴 계정 `mb_id` 원문 노출 여부 재검사
  - 필요하면 장기적으로 `mb_no` 기반 관리자 라우팅 전환 검토
- 완료 기준:
  - 일반적인 관리자 화면에서 탈퇴 회원 원문 `mb_id`가 노출되지 않는다.

### 4. 로컬 QA 환경 정리 정책 확정

- 목표:
  - QA 컨테이너, 테스트 데이터, 임시 계정 정리 기준을 명확히 한다.
- 할 일:
  - 로컬에 남겨둘 컨테이너와 정리할 데이터 범위 결정
  - 테스트 계정 네이밍 규칙과 삭제 시점 정의
  - `.tools/` 및 로컬 전용 보조 자산 커밋 제외 규칙 확인
- 완료 기준:
  - 다음 검증 전후 정리 규칙이 문서로 고정된다.

## 보류 판단

- `mb_id`를 인코딩/복호화하는 방식은 현재 채택하지 않는다.
- 이유:
  - 보호 효과 대비 복잡도가 크다.
  - 현재 코드베이스는 `mb_id`를 직접 키로 사용하는 경로가 많다.
  - 현 단계에서는 `mb_id` 내부 보존 + UI 마스킹이 더 실효적이다.
