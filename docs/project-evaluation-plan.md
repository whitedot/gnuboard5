# 현재 프로젝트 종합 평가 계획

## 1. 목적

이번 평가는 현재의 member-only 리팩토링 결과물이 실제 운영, 유지보수, 다음 단계 확장 논의에 들어갈 만큼 안정적인지 판단하기 위한 것이다.

평가 대상은 코드 품질만이 아니다. 축소된 범위가 런타임 흐름, 관리자 UI, 문서, 빌드 산출물, 운영 절차까지 일관되게 반영되어 있는지도 함께 본다.

## 2. 현재 상태 요약

현재 저장소를 기준으로 보면 다음 특징이 있다.

- 운영 범위가 회원, 인증, 관리자 회원 관리 중심으로 축소되어 있다.
- PHP 레거시 구조는 유지하지만, 회원 도메인 책임은 일부 helper 라이브러리로 분리되어 있다.
- 관리자 UI 스타일은 `tailwind4/` 소스를 기준으로 관리되고 `adm/css/admin.css` 같은 산출물로 빌드된다.
- 수동 QA 문서는 존재하지만, 자동 테스트 기반은 현재 뚜렷하게 확인되지 않는다.
- 수동 QA 문서와 정적 검토 기준은 있으나, 최종 운영 판단은 여전히 사람의 검토에 크게 의존한다.

## 3. 평가 목표

이번 평가는 아래 다섯 질문에 답할 수 있어야 한다.

1. 현재 축소 범위가 숨은 레거시 의존성 없이 일관되게 구현되어 있는가
2. 회원/관리자 핵심 흐름이 실제 운영에 투입 가능한 수준으로 안정적인가
3. 사람이든 AI든 안전하게 수정할 수 있을 만큼 구조가 이해 가능하고 유지보수 가능한가
4. 보안과 운영 점검 체계가 회원 중심 서비스에 필요한 최소 수준을 갖추고 있는가
5. 지금 바로 다음 단계로 넘어가도 되는가, 아니면 안정화 사이클이 먼저 필요한가

## 4. 평가 범위

### 포함 범위

- `member/` 하위 회원 흐름
- `adm/` 하위 관리자 회원/설정 화면
- `lib/` 하위 회원 도메인 공용 라이브러리
- `tailwind4/`, `css/`, `adm/css/` 하위 스타일 소스와 산출물
- member-only 운영에 남겨진 설치, 설정, 런타임 경로
- 범위, QA, 아키텍처를 설명하는 문서

### 제외 범위

- 이미 제거된 커뮤니티/쇼핑 기능 자체의 완성도 평가
- 저장소 제약에 없는 대규모 현대화 논의
- 현재 운영 준비도와 직접 연결되지 않는 대형 디자인 재정비 논의

## 5. 평가 영역과 기준

### A. 범위 정합성

목표: 저장소가 실제로 member-only 시스템처럼 동작하는지 확인한다.

주요 확인 항목:

- 제거된 기능이 활성 화면, 설정 폼, 설치 스크립트, 런타임 helper에서 다시 참조되지 않는가
- 관리자/설정 화면이 제거 대상 기능용 제어 항목을 노출하지 않는가
- 공개/회원 흐름이 더 이상 범위 밖 선택 기능을 전제하지 않는가
- 문서, 코드, 실제 동작 범위가 서로 같은 제품을 설명하고 있는가

근거 자료:

- `README.md`
- `docs/member-only-implementation-plan.md`
- `docs/member-only-scope.md`
- `adm/config_form*.php`
- 설치 스크립트와 공용 유틸리티

### B. 기능 안정성

목표: 핵심 회원 운영 흐름이 처음부터 끝까지 안정적으로 동작하는지 확인한다.

주요 확인 항목:

- 회원가입이 현재 의도한 동의/이메일 인증 흐름으로 정상 동작하는가
- 로그인, 로그아웃, 비밀번호 재설정, 회원정보 수정, 회원탈퇴가 정상 동작하는가
- 관리자 회원 CRUD와 목록 페이징이 제거된 설정 의존성 없이 동작하는가
- 잘못된 입력이나 예외 상황에서 차단과 응답이 예측 가능하게 동작하는가

근거 자료:

- `docs/member-only-remaining-work.md`
- `member/*.php`
- `adm/member_*.php`
- 실제 수동 QA 기록

### C. 아키텍처와 유지보수성

목표: 현재 구조가 수정 위험과 문맥 탐색 비용을 실제로 낮추는지 확인한다.

주요 확인 항목:

- 컨트롤러 파일이 얇게 유지되고 request, validation, persist, flow 조합 역할에 집중하는가
- 회원 관련 책임 분리가 문서화된 파일 경계를 계속 지키고 있는가
- 공용 로직이 다시 대형 혼합 파일로 회귀하고 있지 않은가
- 네이밍, 파일 경계, 문서만으로 수정 지점을 빠르게 찾을 수 있는가

근거 자료:

- `docs/architecture/member-collaboration-phase1.md`
- `lib/member.request.lib.php`
- `lib/member.validation.lib.php`
- `lib/member.persist.lib.php`
- `lib/member.flow.lib.php`
- `member/register_form_update.php`
- `adm/member_form_update.php`

### D. UI 일관성과 빌드 위생

목표: 관리자/사이트 UI 변경이 재현 가능하고 유지보수 가능한지 확인한다.

주요 확인 항목:

- 공용 semantic 클래스가 `tailwind4/common.css` 등 소스 파일에서 관리되는가
- 생성된 CSS가 소스 기준으로 재빌드 가능하며 산출물 드리프트가 없는가
- 관리자 폼 전환 규칙이 남겨진 화면에 일관되게 적용되고 있는가
- 남아 있는 UI 편차가 의도된 것인지, 후속 작업인지 구분되어 있는가

근거 자료:

- `tailwind4/common.css`
- `tailwind4/admin.css`
- `adm/css/admin.css`
- `docs/admin-form-conversion-priority.md`
- `package.json` 빌드 스크립트

### E. 보안과 운영 리스크

목표: 회원 중심 서비스로 운영할 때 감당 가능한 수준의 리스크 관리가 되는지 확인한다.

주요 확인 항목:

- 인증, 비밀번호 재설정 경로가 불일치 입력과 비정상 입력을 제대로 차단하는가
- 이메일 흐름이 현재 의도한 메일만 발송하는가
- 회원정보 수정과 관리자 수정 경로에서 중요한 필드가 조용히 누락되지 않는가
- 운영자가 따라갈 수 있는 보안 점검 절차와 주의사항이 문서화되어 있는가
- 운영자가 알아야 할 민감한 전제가 문서에 남아 있는가

근거 자료:

- `docs/member-only-implementation-plan.md`에 기록된 최근 수정 사항
- `.github/workflows/*`
- 회원/관리자 저장 경로
- validation/persist 로직 검토 결과

### F. 문서와 협업 준비도

목표: 다음 작업자가 매번 문맥을 다시 복원하지 않고도 이어서 작업할 수 있는지 확인한다.

주요 확인 항목:

- README, 범위 문서, QA 체크리스트, 아키텍처 문서가 서로 충돌하지 않는가
- 새 협업자가 활성 범위, 빌드 방법, QA 절차를 빠르게 파악할 수 있는가
- 남은 일과 보류 리스크가 명확히 기록되어 있는가
- 사람/AI 협업을 고려한 구조와 문서 기준이 실제로 반영되어 있는가

근거 자료:

- `README.md`
- `docs/*.md`
- `AGENTS.md`

## 6. 점수화 방식

각 평가 영역은 5점 척도로 본다.

- 5점: 바로 다음 단계로 넘어가도 무리가 없는 상태
- 4점: 전반적으로 양호하나 소규모 보정이 필요한 상태
- 3점: 사용 가능하지만 의미 있는 약점이 남은 상태
- 2점: 불안정하거나 정합성이 낮아 우선 보정이 필요한 상태
- 1점: 현재 목표 기준으로 수용하기 어려운 상태

권장 가중치는 아래와 같다.

- 범위 정합성: 20%
- 기능 안정성: 25%
- 아키텍처와 유지보수성: 20%
- UI 일관성과 빌드 위생: 10%
- 보안과 운영 리스크: 15%
- 문서와 협업 준비도: 10%

종합 판단 기준:

- 4.2~5.0: 다음 단계 계획 수립 가능
- 3.5~4.1: 소규모 보정 후 조건부 진행 가능
- 2.5~3.4: 안정화 사이클 선행 필요
- 2.5 미만: 범위 또는 품질 기준 재정의 필요

## 7. 수행 절차

### 1단계. 문서/범위 검토

- README, 범위 문서, QA 문서, 아키텍처 문서를 서로 대조한다.
- 문서상 선언된 범위와 실제 저장소 구조 사이의 불일치를 기록한다.
- 평가 시 집중해서 볼 가설과 리스크 포인트를 1차로 정리한다.

### 2단계. 정적 코드 리뷰

- 회원/관리자 컨트롤러가 문서화된 계층 규칙을 지키는지 확인한다.
- 공용 회원 라이브러리에 책임 누수나 숨은 결합이 없는지 확인한다.
- 설정/설치/런타임 파일에 제거 대상 기능 참조가 남아 있는지 본다.
- Tailwind 소스와 산출물 흐름이 재현 가능한지 확인한다.

### 3단계. 수동 시나리오 검증

- 기존 member-only QA 체크리스트를 실제로 실행한다.
- 테스트 일시, 브랜치, 커밋 해시, 실패 단계, 실제 결과, 재현 조건을 기록한다.
- 이메일/인증 흐름은 가능하면 스크린샷 또는 메일 샘플까지 남긴다.

### 4단계. 운영/보안 검토

- 운영 전후에 필요한 보안 점검 절차와 주의사항을 확인한다.
- 비밀번호 재설정, 이메일 인증, 관리자 회원 수정, 회원탈퇴 같은 고위험 경로를 다시 점검한다.
- 열린 이슈를 심각도와 운영 영향 기준으로 분류한다.

### 5단계. 종합 판단

- 각 평가 영역에 점수를 부여한다.
- 치명적 차단 요소, 허용 가능한 리스크, 후순위 개선 과제를 구분한다.
- 최종 결론을 아래 셋 중 하나로 정리한다.

1. 다음 단계로 진행
2. 현재 단계 안정화 우선
3. 범위를 더 좁힌 뒤 재평가

## 8. 산출물

평가 한 사이클이 끝나면 아래 산출물이 남아야 한다.

- 종합 평가 요약 보고서
- 영역별 점수표
- 결함/리스크 목록과 우선순위
- 수동 QA 실행 기록
- 향후 2~4주 권장 작업 로드맵

## 9. 권장 진행 방식

실무적으로는 아래 순서가 가장 효율적이다.

1. 문서 정합성 검토
2. 회원/관리자 핵심 경로 정적 리뷰
3. 수동 QA 실행
4. 보안/운영 점검 절차 확인
5. 점수화와 진행 여부 결정

권장 역할 분담:

- 제품/범위 검토: 프로젝트 오너
- 구조/리스크 리뷰: 개발자 또는 AI 보조 리뷰어
- 수동 시나리오 실행: 운영자 또는 관리자 계정 보유 리뷰어
- 최종 점수화와 우선순위 결정: 오너 + 리뷰어 공동

## 10. 현재 저장소 기준 우선 점검 포인트

이번 저장소에서는 특히 아래 다섯 가지를 먼저 보는 것이 좋다.

- 문서상 남은 핵심 작업인 수동 QA가 실제로 실행되고 기록되었는가
- 제거한 레거시 기능 참조가 관리자/설정/설치/런타임 경로에 다시 스며들지 않았는가
- 회원 도메인 책임 분리가 최신 컨트롤러와 라이브러리에서도 유지되는가
- Tailwind 소스 기준과 생성된 관리자 CSS가 계속 동기화되고 있는가
- 자동 테스트 부재가 다음 변경 사이클에서 과도한 회귀 위험을 만들고 있지 않은가

## 11. 현재 단계 통과 기준

현재 단계는 아래 조건을 모두 만족할 때만 완료로 본다.

1. 핵심 회원 시나리오가 수동 QA를 통과했고 근거 기록이 남아 있다.
2. 회원/인증/관리자 회원 흐름에 P1 또는 미해결 운영성 P2 이슈가 남아 있지 않다.
3. 제거된 기능이 설정, 설치, 런타임 참조로 다시 드러나지 않는다.
4. 공용 스타일 변경이 Tailwind 소스 기준으로 재빌드 가능하고 수동 산출물 편집에 의존하지 않는다.
5. 프로젝트 문서가 현재 범위와 남은 일을 같은 기준으로 설명한다.
